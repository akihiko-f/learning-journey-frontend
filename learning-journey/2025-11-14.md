## 学習メモ

### リポジトリ再構成の目的

**変更前の構造：**
```
learning-journey-frontend/
├── todo-app/
├── timer-app/
├── budget-app/
├── calculator-app/
├── memo-app/
├── favorites-app/
├── free-practice-app/
└── learning-journey/
```

**変更後の構造：**
```
learning-journey/
├── frontend/
│   ├── todo-app/
│   ├── timer-app/
│   ├── budget-app/
│   ├── calculator-app/
│   ├── memo-app/
│   ├── favorites-app/
│   └── free-practice-app/
└── learning-journey/
```

### 再構成の利点

1. **明確な意図の表現**
   - リポジトリ名が学習の旅（learning-journey）そのものを表現
   - frontendはその一部であることが構造から分かる

2. **拡張性の向上**
   - 将来的に`backend/`や`mobile/`などを追加しやすい
   - 各技術領域を独立したディレクトリで管理できる

3. **整理されたプロジェクト構造**
   - ルート直下がすっきりし、全体像が把握しやすい
   - 練習アプリが一箇所にまとまり、管理が容易

4. **学習記録との分離**
   - コード（frontend/）と学習記録（learning-journey/）が明確に分かれる
   - それぞれの目的が明確になる

### Git履歴の保持

`git mv`コマンドを使用することで：
- ファイルの移動履歴がGitに記録される
- 過去のコミット履歴が失われない
- `git log --follow`でファイルの移動を追跡可能

### モノレポ的アプローチ

この構造は「モノレポ（monorepo）」の考え方に近い：
- 複数のプロジェクト（アプリ）を一つのリポジトリで管理
- 共通の設定や学習記録を一箇所で管理
- プロジェクト間の関連性が明確

### リポジトリ構成の設計思想

この再構成によって、以下のような将来の展開が可能になります：

```
learning-journey/
├── frontend/           # フロントエンド練習
│   ├── todo-app/
│   └── ...
├── backend/            # 将来的にバックエンド学習を追加
│   ├── api-server/
│   └── ...
├── mobile/             # モバイルアプリ学習
│   └── ...
└── learning-journey/   # 全ての学習記録
    ├── 2025-10-16.md
    └── ...
```
